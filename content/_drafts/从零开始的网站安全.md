## 身份验证概览-AI总结
通过一个**用户从输入网址到进入个人主页**的完整生命周期，可以透彻理解这些概念的物理位置与逻辑职责。

### 1. 登录阶段：凭证转换 (Credentials to Token)
当你在登录框输入账号密码点击“登录”时：
* **动作**：浏览器发起一个 `POST /login` 请求。
* **服务器处理**：验证数据库，确认你是张三。
* **签发令牌 (Token)**：服务器不能直接把密码还给你，它会生成一个“通行证”。这个通行证的**统称**就是 **Token**（令牌）。
* **具体实现**：如果服务器使用 **JWT** 格式，它会将“张三、普通用户、2026年过期”等信息用私钥签名，生成一段字符串。

### 2. 存储阶段：令牌的家 (Where to stay)
服务器把这个 Token 发回给浏览器后，浏览器需要找个地方存起来，否则刷新页面就丢了。
* **Cookie**：这是浏览器的一种**自动化存储机制**。如果服务器在响应头里写了 `Set-Cookie: token=abc...`，浏览器会自动把它存在磁盘或内存里。
* **Session**（传统模式）：服务器不在 Token 里写信息，而是只发一个 `SessionID`。服务器自己在内存（Redis）里记账：`SessionID 001 = 张三`。
* **本地存储 (LocalStorage)**：现代网页常把 JWT 存在这里，由 JavaScript 代码手动管理。



### 3. 请求携带阶段：亮出证件 (Bearer)
当你点击“我的订单”页面时，浏览器必须证明你是张三。
* **传输协议 (Bearer)**：这是一种**格式规范**。就像你进考场不仅要带准考证，还要放在透明袋里。在 HTTP 请求头中，你会看到：
    `Authorization: Bearer <你的JWT字符串>`
    这里的 **Bearer** 告诉服务器：“后面跟着的是一个合法的持有者令牌，请解析它。”
* **自动驾驶 (Cookie 模式)**：如果你使用的是 Cookie 存储，浏览器会自动在 Header 里带上 `Cookie: sessionid=001`，不需要开发者写代码。

### 4. 验证阶段：审查流程 (The Verification)
服务器拿到请求后，开始“安检”：
* **如果是 Session**：服务器拿着 `SessionID` 去查自己的账本（Redis）。如果账本里有，通过。
* **如果是 JWT**：服务器不查账本。它直接用自己的公钥/私钥去算这段字符串的签名。如果数学计算正确，说明没人改过，直接从字符串里读出“张三”，通过。







由于HTTP本身是无状态的,也就是说服务器无法通过协议本身的字段来保留用户信息,那么服务器就有必要针对所有的网络请求都进行审查,从而过滤掉恶意请求.

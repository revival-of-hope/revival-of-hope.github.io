## 为什么要有数据库

在计算机发展的早期，数据是直接存储在**文本文件**或**二进制文件**中的。随着数据量的增长，直接操作文件暴露了三个致命缺陷：

* **数据冗余与不一致**：同一个信息（如客户地址）可能存在于多个文件中，修改时极易漏改，导致数据冲突。
* **访问困难**：程序员必须为每一种查询需求编写特定的解析代码，缺乏通用性。
* **缺乏并发控制**：当多个人同时修改同一个文件时，后保存的人会覆盖前者的修改，导致数据丢失。


## 数据库管理系统的出现

为了摆脱“手动操作文件”的低效，**数据库管理系统（DBMS, Database Management System）**应运而生。它是一个位于用户与操作系统文件之间的**软件层**。

* **历史起点**：20世纪60年代，为了阿波罗登月计划，IBM开发了第一个层次模型数据库 IMS。随后，网状模型出现。
* **核心逻辑**：DBMS 将物理存储（文件怎么放）与逻辑模型（数据看起来像什么）解耦。用户不再直接操作文件，而是向 DBMS 发送指令，由 DBMS 负责索引维护、事务管理和崩溃恢复。

## SQL的出现

早期数据库（层次或网状）的操作极其复杂，查询数据像是在迷宫里导航，必须指明“如何去取”数据的路径。

* **关系模型理论**：1970年，IBM 研究员 E.F. Codd 发表论文，提出将数据组织成**二维表**。
* **声明式语言**：为了配合关系模型，IBM 开发了 Structured Query Language (SQL)。
* **历史意义**：SQL 的核心在于“**声明式**”——你只需要告诉数据库“我要什么”（What），而不需要告诉它“怎么去拿”（How）。这极大地降低了数据操作的门槛，使得关系型数据库在 80 年代后统治了市场。

## 常见数据库管理系统种类

随着应用场景的多样化，数据库演化出了多种形态：

| 种类                        | 代表产品                              | 特点与缘由                                                       |
| --------------------------- | ------------------------------------- | ---------------------------------------------------------------- |
| **关系型数据库 (RDBMS)**    | MySQL, PostgreSQL, Oracle, SQL Server | 基于表结构，支持强一致性和复杂的 SQL 查询。历史最久，生态最稳。  |
| **键值存储 (Key-Value)**    | Redis, Memcached                      | 极高性能的内存数据库。为了解决 Web 2.0 时代的高并发缓存需求。    |
| **文档型数据库 (Document)** | MongoDB                               | 存储类似 JSON 的文档。为了解决半结构化数据和模式灵活变动的需求。 |
| **列式数据库 (Columnar)**   | ClickHouse, HBase                     | 按列存储。为了大数据量下的聚合分析（OLAP）而设计。               |
| **图形数据库 (Graph)**      | Neo4j                                 | 以点和边存储。为了处理社交网络、推荐系统等复杂关联关系。         |
## 总结
到了现在,主流互联网公司用的数据库不外乎这三种:
1. Redis: 用于控制缓存和高并发
2. MySQL: 保证数据安全和高可用
3. MongoDB: 存储非结构化数据和媒体信息.
